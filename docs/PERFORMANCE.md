# Производительность Mehmet Metrics

## Текущий контракт

Проект оптимизирован под небольшой, редко меняющийся внутренний набор данных:
JSON обновляется перед деплоем, а посетители получают заранее собранные страницы.

- `/`, `/catalog`, `/money` и `/work` должны отображаться как `○ Static` в
  выводе `next build`.
- `/product/[id]` должен отображаться как `● SSG`; все product id создаются
  через `generateStaticParams`, неизвестные id не рендерятся динамически.
- `app/layout.tsx` не использует `headers()`, `cookies()` и другие request-bound
  API.
- Generated JSON импортируют серверные data-модули. Клиентские компоненты
  получают сериализованные props и не импортируют `data/*.json`.
- Закрытые годы и поставки Work не монтируют внутренние таблицы.

## Изображения

Один исходный JPG имеет два производных варианта:

```text
jpg/<имя>.jpg        источник и последний fallback
webp/<имя>.webp      детальная страница товара
webp/card/<имя>.webp главная и карточки каталога
```

Карточный вариант ограничен размером `960 × 1200`, не увеличивает маленькие
исходники и сохраняется с quality 80. Полный вариант сохраняется с quality 85.
`scripts/convert_to_webp.py` пересобирает отсутствующий или изменённый вариант.
Он также рекурсивно удаляет `.webp`, которым больше не соответствует исходный
JPG/JPEG; другие расширения не затрагиваются.

Цепочка загрузки карточки:

```text
card WebP → full WebP → исходный JPG/JPEG → __photo_pending → 📷
```

Первая карточка главной и основное фото товара имеют `priority`; остальные
изображения остаются lazy.

## Контрольный замер

Замер выполнен production-сборкой 24 июля 2026 года.

| Маршрут | До: рендер | После: рендер | До: First Load JS | После: First Load JS |
| --- | --- | --- | ---: | ---: |
| `/` | Dynamic | Static | 110 kB | 109 kB |
| `/catalog` | Dynamic | Static | 114 kB | 111 kB |
| `/money` | Dynamic | Static | 113 kB | 104 kB |
| `/product/[id]` | Dynamic | SSG | 112 kB | 110 kB |
| `/work` | Dynamic | Static | 117 kB | 109 kB |

Набор из 58 полноразмерных WebP занимает 5,02 МБ. Карточные варианты тех же
файлов занимают 2,16 МБ — на 57% меньше; самый тяжёлый карточный файл —
128,3 КБ.

Эти числа — контрольная точка, а не жёсткий запрет на небольшие колебания после
обновления Next.js или данных. Важные регрессии: появление `ƒ Dynamic`,
возвращение JSON в client chunks и использование полноразмерных фото в сетках.

## Как повторить проверку

1. Остановить dev server, чтобы он не удерживал `.next/trace`.
2. Запустить:

   ```bash
   npm run validate:images
   npm run build
   ```

3. Проверить, что маршруты остались Static/SSG и First Load JS не вырос без
   объяснимой причины.
4. После изменения исходных JPG выполнить:

   ```bash
   python scripts/convert_to_webp.py --auto
   npm run validate:images
   ```

   При удалении или переименовании JPG эта же команда очищает оба производных
   варианта. Если изменился сам набор фото каталога, перед ней нужно запустить
   Excel-парсер; стандартный сценарий «Запустить с обновлением» уже соблюдает
   этот порядок.

5. Перед деплоем запустить `npm run preflight`.

Виртуализация списков сейчас не нужна: текущий объём — 44 поставки и 117
позиций, а закрытые секции не монтируются. Вернуться к ней стоит, если один
раскрытый год начнёт содержать сотни одновременно отображаемых строк и
профилировщик покажет проблему рендера.
