# Excel → JSON pipeline

## Источник и результат

Источник правды для поставок, статусов, размеров, материалов и актуальных цен —
Excel / Google Sheet, лист «Поставки».

Пайплайн создаёт:

- `data/shipments.json`;
- `data/products.json`;
- `data/meta.json`.

`data/money.json` не генерируется и валидируется отдельно как ручной источник.

## Точки входа

| Команда | Назначение |
| --- | --- |
| `python Excel/fetch_google_sheet.py` | Скачать актуальный XLSX |
| `python Excel/parse_excel.py --auto` | Распарсить Excel, обновить каталог, проверить и записать JSON |
| `python Excel/update_prices.py` | Повторно синхронизировать price/cost каталога из существующих поставок |
| `python Excel/validate_generated_data.py` | Проверить generated JSON и `money.json` |
| `python Excel/test_parser_logic.py` | Запустить регрессионные тесты парсера |

Windows-сценарий `Запустить с обновлением.bat` выполняет fetch → parse → WebP →
fast preflight → dev server.

## Модули

- `parse_excel.py` — оркестратор полного импорта;
- `excel_parser.py` — чтение листа и построение поставок;
- `parser_utils.py` — разбор названий, размеров, материалов, категорий и фото;
- `catalog_pricing.py` — выбор последних price/cost;
- `data_validator.py` — проверка структуры и доменных инвариантов;
- `json_storage.py` — чтение и атомарная запись JSON;
- `fetch_google_sheet.py` — загрузка XLSX;
- `update_prices.py` — ремонтная актуализация цен;
- `validate_generated_data.py` — standalone-валидация;
- `test_parser_logic.py` — Python-регрессии.

## Используемые колонки

| Колонка | Значение | Результат |
| --- | --- | --- |
| A | № поставки / разделитель года | `shipment.id`, границы поставки, `year` |
| C | Наименование | товар, вариант, размеры |
| D | Состав | `product.materials` |
| E | Статус позиции | `rawItem.status` |
| F | Статус поставки | `shipment.status` |
| G | Количество | `quantityOverride`, когда оно задано |
| H | Цена за единицу, $ | `rawItem.price` |
| J | Курс списания | guard для себестоимости |
| N | Себестоимость с карго, ₽ | `rawItem.cost`, только при положительном J |
| P | Дата поступления | `receivedDate` или `eta` |

Остальные колонки не должны неявно становиться источником новых полей без
изменения валидатора, тестов и документации.

## Границы поставок и годы

- строка с годом в колонке A и без содержательной позиции задаёт текущий год;
- строка с номером поставки и наименованием начинает новую поставку;
- следующие строки с наименованием и без нового номера относятся к текущей
  поставке;
- новый номер, новый разделитель года или конец данных закрывает поставку;
- разделитель года не попадает в `rawItems`.

## Наименования и productId

Размеры читаются из последней пары скобок в наименовании. Имя товара очищается
от размерного суффикса и сопоставляется с каталогом.

- существующий товар сохраняет свой `productId`;
- новый товар получает стабильный id вида `auto-NNN`;
- `excelRows` хранит все строки источника для диагностики;
- каталог после полного парсинга содержит только товары из текущего источника.

## Размеры и количество

Канонические ключи raw data:

```text
xs, s, m, l, xl, OneSize
```

Неизвестный или повторно указанный размер — ошибка.

Приоритет количества:

1. явное значение колонки G (`quantityOverride`);
2. сумма размеров;
3. legacy fallback для одиночного sample без другого количества.

Маркер `(на уточнении)` создаёт `sizesUnknown: true`; количество берётся из
колонки G, а фиктивный размер в каталог не добавляется.

`sample` — маркер, а не принудительное количество `1`.

## Категории

Допустимы только:

- `Мех`;
- `Замша`;
- `Кожа`;
- `Экзотика`.

Приоритет определения: Экзотика → Кожа → Мех → Замша. Неизвестная категория
останавливает импорт; fallback `Прочее` запрещён.

## Цена и себестоимость

- историческая цена хранится в позиции поставки;
- каталог получает последнюю доступную price/cost по хронологии поставок;
- значение N импортируется как `cost` только при положительном курсе J;
- объединённая ячейка J действует на покрываемые ею строки;
- пустой, нулевой или отрицательный J не позволяет cargo-only формуле из N
  стать реальной себестоимостью.

## Статусы и оплата

Текст статуса сохраняется из Excel без замены на enum.

На стороне TypeScript `isPaidStatus()` считает статус оплаченным, только если
есть положительная формулировка «оплачен» и отсутствуют отрицания или частичная
оплата. Пустой статус позиции наследует статус поставки.

`paidPreviously` и `noPayment` делают позицию неплатёжной через `isPayable`.
Ручные исключения не следует прятать в статусный текст.

## Фото

Фото опционально. Поле `photo` создаётся только при точном JPG/JPEG в
`public/images/products/jpg/`.

- отсутствие `photo` валидно и показывает `__photo_pending`;
- заданный путь к отсутствующему файлу — ошибка;
- `scripts/convert_to_webp.py` создаёт полноразмерный `webp/` и уменьшенный
  `webp/card/`, обновляет изменённые варианты и рекурсивно удаляет производные
  `.webp`, для которых больше нет JPG/JPEG;
- `npm run validate:images` проверяет ссылки, JPG, оба WebP-варианта и лишние
  файлы.

Папка `jpg/` является источником правды. Полный update-flow сначала пересобирает
`products.json` из Excel и актуального списка JPG, а затем синхронизирует WebP.
Поэтому для удаления фото достаточно удалить исходник и запустить
«Запустить с обновлением».

## Безопасная запись

Полный импорт сначала собирает данные в памяти, затем:

1. актуализирует каталог;
2. проверяет shipments/products/meta;
3. при успехе атомарно заменяет JSON.

Ошибка парсинга или валидации не должна оставлять частично обновлённый набор
generated data.

## Обязательные проверки после изменения пайплайна

```bash
npm run test:excel
npm run validate:data
npm run validate:images
npm run test:images
npm run test
npm run typecheck:strict
```

Перед деплоем выполняется `npm run preflight`.
