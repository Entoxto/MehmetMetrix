ЛОГИКА ПАРСИНГА EXCEL → JSON
================================

Источник: Расчёты с мехметом new.xlsx
Лист: Поставки
Целевой файл: data/shipments.json

ОБЩАЯ СТРУКТУРА EXCEL:
- 101 строка, 17 колонок
- Заголовок в строке 0 (индекс 8 в preview)
- Данные начинаются со строки 1 (индекс 9)
- Разделители годов (2024, 2025) - строки, где в колонке A (№ Поставки) указан только год (например, "2024" или "2025"), а остальные колонки пустые или почти пустые

СТРУКТУРА ДАННЫХ:
- Каждая поставка начинается со строки, содержащей одновременно № Поставки (колонка A / индекс 0) И Наименование (колонка C / индекс 2)
- Все последующие строки, где № Поставки нет, но есть Наименование - это позиции текущей поставки
- Поставка заканчивается когда:
  * Встречаем новую строку с № Поставки И Наименование (начало следующей поставки)
  * Встречаем разделитель года (строка с годом, например "2024" или "2025")
  * Достигаем пустой строки (конец данных)
- Статусы поставок указываются в колонке F (индекс 5), обычно на первой строке поставки
- Пустые строки (71-100 в preview) - игнорируем

ОПРЕДЕЛЕНИЕ ГОДА ПОСТАВКИ:
- Разделители годов - это строки, где в колонке A (№ Поставки) указан только год (например, "2024" или "2025")
- Все поставки, которые идут ПОСЛЕ разделителя года, относятся к этому году
- Год определяется для каждой поставки на основе последнего встреченного разделителя года
- Алгоритм:
  1. При парсинге строк Excel проходим сверху вниз
  2. При встрече строки, где колонка A содержит только год (например, "2024" или "2025") - сохраняем этот год как текущий
  3. Все последующие поставки (до следующего разделителя года) получают этот год
  4. Если разделитель года встречен в середине поставки - поставка завершается, следующий разделитель определяет год для новых поставок
  5. Год сохраняется в поле `year` в JSON (см. раздел 4. МАППИНГ ПОЛЕЙ ПОСТАВКИ)
  
  ПРИМЕРЫ из Excel:
  - Строка 1 (индекс 1): A="2024", остальные колонки пустые → разделитель года 2024
  - Строки 2-18: поставки относятся к 2024 году
  - Строка 19 (индекс 19): A="2025", остальные колонки пустые → разделитель года 2025
  - Строки 20-70: поставки относятся к 2025 году
  
  ВАЖНО: Разделитель года НЕ является поставкой и НЕ добавляется в rawItems. Это только маркер для определения года последующих поставок.

ОБРАБОТКА:

1. ОПРЕДЕЛЕНИЕ ГРАНИЦ ПОСТАВКИ:
   ПРАВИЛО: Поставка начинается в строке, где одновременно присутствуют:
   - № Поставки (колонка A) - заполнено
   - Наименование (колонка C) - заполнено
   
   Все последующие строки, где:
   - № Поставки пустое ИЛИ отсутствует
   - НО Наименование заполнено
   
   → относятся к той же поставке
   
   Поставка заканчивается когда:
   - Встречаем строку с № Поставки И Наименование (начало новой поставки)
   - Встречаем разделитель года (строка с годом без других данных)
   - Достигаем пустой строки (конец данных)
   
   ПРИМЕР из Excel (строки 2-5 в preview):
   Строка 2: № Поставки=1, Наименование="Плащ из замши..." → НАЧАЛО ПОСТАВКИ 1 → позиция поставки 1
   Строка 3: № Поставки=пусто, Наименование="Юбка из замши..." → позиция поставки 1
   Строка 4: № Поставки=пусто, Наименование="Жакет из замши..." → позиция поставки 1
   Строка 5: № Поставки=пусто, Наименование="Жакет из кожи..." → позиция поставки 1
   Строка 6: № Поставки=2, Наименование="Дублёнка длинная..." → НАЧАЛО ПОСТАВКИ 2 → позиция поставки 2
   
   Таким образом, поставка 1 состоит из строк 2-5, поставка 2 начинается со строки 6.

2. ИГНОРИРУЕМ:
   - Строки с пустым № Поставки И пустым Наименованием:
     * Разделители годов (например, строка с текстом "2024" или "2025")
     * Полностью пустые строки (после строки 70 в preview)
   - Строки где все значимые поля пустые
   
   ВАЖНО: Строка с пустым № Поставки, но с заполненным Наименованием - НЕ игнорируется,
   она относится к текущей поставке (смотри правило 1).

3. ГРУППИРОВКА ПО ПОСТАВКАМ:
   Алгоритм:
   1. Инициализируем переменную currentYear = null (или текущий год по умолчанию)
   2. Проходим по строкам сверху вниз
   3. При встрече строки-разделителя года (колонка A содержит только год, например "2024" или "2025"):
      - Обновляем currentYear = значение из колонки A (преобразуем в число)
      - Пропускаем эту строку (не добавляем в поставки)
      - Продолжаем обработку со следующей строки
   4. При встрече строки с № Поставки И Наименование - начинаем новую поставку
   5. Сохраняем номер поставки как текущую
   6. Сохраняем currentYear как год этой поставки
   7. Первую строку (с № Поставки И Наименование) добавляем как позицию поставки
   8. Все следующие строки с Наименованием (но без № Поставки) добавляем к текущей поставке
   9. Прекращаем добавление к поставке при встрече:
      - Строки с № Поставки И Наименование (начало новой поставки) → завершаем текущую поставку, переходим к шагу 4
      - Разделителя года (строка с годом без других данных) → завершаем текущую поставку, обновляем currentYear, переходим к шагу 3
      - Пустой строки → завершаем текущую поставку, завершаем обработку
   10. После завершения поставки (перед переходом к следующей):
      - Устанавливаем year = currentYear для этой поставки
      - Собираем все значения из колонки P по всем строкам этой поставки
      - Определяем receivedDate или eta согласно правилам (см. раздел 4. МАППИНГ ПОЛЕЙ ПОСТАВКИ)
   11. Повторяем для следующей поставки
   12. После обработки всех поставок:
      - Сортируем поставки по году (по убыванию), затем по номеру поставки (по убыванию)
      - Самые новые поставки (с большим годом и номером) будут первыми в списке
      - Пример: поставка №12 2025 года будет перед поставкой №1 2025 года, которая будет перед поставкой №6 2024 года
   
   ВАЖНО: Первая строка поставки (с № Поставки) является одновременно и позицией этой поставки!
   
   Первая строка поставки (с № Поставки) может содержать:
   * № Поставки (колонка A) - используется для идентификации поставки
   * Наименование (колонка C) - это также позиция поставки
   * Дата размещения заказа (колонка B) - игнорируется (не используется)
   * Состав (колонка D) - игнорируется (не используется)
   * Статус поставки (колонка F / индекс 5) - указывается обычно на первой строке
   * Дата поступления продукции / ETA (колонка P):
     - Если значение в формате даты (например "02.10.2024") → receivedDate, отображается как "Получено и дата"
     - Если значение - текст (например "Ожидаем отправку на этой неделе") → eta, отображается как "План поставки: {текст}"
   * Внутри поставки может быть несколько строк с датами получения/ETA, относящиеся к ней
   * ЛОГИКА ОБРАБОТКИ колонки P по всем строкам поставки:
     - Собираем все значения из колонки P по всем строкам поставки
     - Если есть хотя бы одно текстовое значение (ETA) → приоритет у текста, устанавливаем eta
     - Если есть только даты → берём самую позднюю (наиболее актуальную) дату, устанавливаем receivedDate
     - Если дата конфликтует с текстом - берём текст (ETA имеет приоритет)

4. МАППИНГ ПОЛЕЙ ПОСТАВКИ:
   - id: "shipment-{номер_поставки}"
   - title: "Поставка №{номер_поставки}"
   - year: год поставки, определенный по последнему встреченному разделителю года (см. раздел "ОПРЕДЕЛЕНИЕ ГОДА ПОСТАВКИ")
     * Если разделитель года не встречен до поставки → используем текущий год (2025) или определяем из receivedDate
     * Год сохраняется как число (например, 2024, 2025)
   - status: из колонки "Статусы поставок" (колонка F)
     * ТЕКСТ СТАТУСА ПРОКИДЫВАЕТСЯ КАК ЕСТЬ (без маппинга в коды)
     * Примеры: "Получено, оплачено ✅", "Получено, не оплачено 📦", "В работе 🧵" и т.д.
     * Если статус пустой, используется значение по умолчанию "В работе 🧵"
     * Логика "оплачен / не оплачен" определяется на стороне TypeScript функцией isPaidStatus()
   - receivedDate: из колонки P, собираем по ВСЕМ строкам поставки
     АЛГОРИТМ:
     1. Проходим по всем строкам поставки, собираем значения из колонки P
     2. Фильтруем только даты:
        * Если значение - datetime объект (например, "2024-10-02 00:00:00") → преобразуем в формат "DD.MM.YYYY" (например, "02.10.2024")
        * Если значение уже в формате "DD.MM.YYYY" (например, "02.10.2024") → используем как есть
        * Если значение - текст (не дата) → пропускаем
     3. Если есть только даты (нет текста) → выбираем самую позднюю дату
     4. Устанавливаем receivedDate в формате "DD.MM.YYYY"
     5. В приложении отображается "Получено" и дата
     
   - eta: из колонки P, собираем по ВСЕМ строкам поставки
     АЛГОРИТМ:
     1. Проходим по всем строкам поставки, собираем значения из колонки P
     2. Ищем текстовые значения (не даты), например "Ожидаем отправку на этой неделе"
     3. Если есть хотя бы одно текстовое значение → приоритет у текста, устанавливаем eta
     4. Убираем переносы строк (\n) из текста, заменяя их на пробелы
        * Пример: "Ожидаем отправку\nна этой неделе" → "Ожидаем отправку на этой неделе"
     5. В приложении отображается как "План поставки: {текст из eta}"
     
   - ПРИОРИТЕТ: ETA (текст) имеет приоритет над датой. Если в поставке есть и текст, и даты - используется только eta.
   - groupByPayment: определяется автоматически или вручную
     * Автоматически: устанавливаем groupByPayment: true, если ВСЕ позиции в поставке не имеют цены (price отсутствует или null)
     * Вручную: в остальных случаях устанавливается вручную в JSON после парсинга
     * По умолчанию: false (или не устанавливается, если не требуется)
     * Пример: если все позиции без цены → groupByPayment: true

5. МАППИНГ ПОЛЕЙ ПОЗИЦИЙ (rawItems):
   - productId: нужно найти по "Наименование" в products.json (см. раздел 9. МАППИНГ PRODUCTID)
   - overrideName: из колонки "Наименование" (колонка C / индекс 2)
   - price: из колонки "Стоймость 1 ед $" (колонка H / индекс 7) - цена в долларах - если пусто, цена не устанавливается (прочерк) - см. раздел 7.2
   - cost: из колонки "Себестоимость с учётом карго" (колонка N / индекс 13) - себестоимость в рублях - если пусто, себестоимость не устанавливается (прочерк) - см. раздел 7.3
   - sizes: парсим из колонки "Наименование" (колонка C / индекс 2) - см. раздел 8
     * Формат: "(XS-5, S-7, M-5)" или "(one size-5)" или "(образец)"
     * "one size" → "OneSize"
     * "образец" → sample: true, quantityOverride: 1
   - quantityOverride: из колонки "Кол-во в заказе" (колонка G / индекс 6) - см. раздел 7.1
     * Используется если количество отличается от суммы sizes или если размеры не указаны
   - status: из колонки "Статусы позиций" (колонка E / индекс 4)
     * ТЕКСТ СТАТУСА ПРОКИДЫВАЕТСЯ КАК ЕСТЬ (без маппинга в коды)
     * Примеры: "Получено, оплачено ✅", "Получено, не оплачено 📦", "В производстве 🛠️" и т.д.
     * Логика "оплачен / не оплачен" определяется на стороне TypeScript функцией isPaidStatus()
     * isPaidStatus проверяет наличие слова "оплачен" в тексте статуса
   
   - inTransit: true если в колонке E (Статусы позиций) содержится "в пути" (без учёта регистра)
     * Устанавливается автоматически парсером при обнаружении "в пути" в тексте статуса
   - sample: true если в названии есть "(образец)" или в колонке "Кол-во в заказе" = 1 и указано "образец"
   - note: дополнительные примечания (если есть специальные метки в данных)
   
   ДЕТАЛЬНОЕ ОПИСАНИЕ парсинга количества и цены - см. раздел 7

6. ОСОБЫЕ СЛУЧАИ:
   - Первая строка поставки (с № Поставки) также является позицией этой поставки и добавляется в rawItems
   - Если в названии есть "(образец)" - ставим sample: true
   - Обработка колонки P (индекс 15) - собираем по ВСЕМ строкам поставки:
     * Проходим по всем строкам поставки, собираем значения из колонки P
     * Приоритет: ETA (текст) > дата
     * Если есть хотя бы одно текстовое значение → устанавливаем eta, отображается как "План поставки"
     * Если есть только даты → выбираем самую позднюю (наиболее актуальную) дату, устанавливаем receivedDate
     * Дата и ETA не могут быть одновременно - если есть текст, используется только eta
     
     ПРИМЕРЫ:
     - Строка 1: P = "02.10.2024", Строка 2: P = "05.10.2024" → receivedDate = "05.10.2024" (самая поздняя)
     - Строка 1: P = "02.10.2024", Строка 2: P = "Ожидаем отправку" → eta = "Ожидаем отправку" (текст имеет приоритет)
     - Строка 1: P = пусто, Строка 2: P = "05.10.2024" → receivedDate = "05.10.2024"
   - Если статус позиции отличается от статуса поставки - используем статус позиции
   - Дата размещения заказа (колонка B) - игнорируется (не используется в парсинге)
   - Пустые значения в числовых полях:
     * Если значение пустое (null, пустая строка, NaN) → поле не устанавливается в JSON
     * Не записываем null, 0 или пустую строку - просто пропускаем поле
     * Примеры:
       - price: пусто → поле price отсутствует в JSON
       - quantityOverride: пусто → поле quantityOverride отсутствует в JSON

7. ПАРСИНГ КОЛИЧЕСТВА И ЦЕНЫ:
   
   7.1. КОЛИЧЕСТВО В ЗАКАЗЕ (колонка G):
   - Берём значение из колонки "Кол-во в заказе" (колонка G / индекс 6)
   - Это общее количество единиц данной позиции в поставке
   - Может быть числом (например, 5, 17, 20)
   - Используется для:
     * quantityOverride - если количество в колонке G отличается от суммы по размерам
     * Проверки соответствия с количеством, вычисленным из размеров
   
   ЛОГИКА:
   - Если количество из колонки G совпадает с суммой размеров из названия → используем sizes
   - Если количество из колонки G отличается от суммы размеров → устанавливаем quantityOverride
   - Если в названии нет размеров, но есть количество в колонке G → используем quantityOverride
   
   ПРИМЕРЫ:
   - Название: "Плащ из замши — коричневый (one size-5)", Колонка G: 5
     → sizes: {OneSize: 5}, quantityOverride: не устанавливаем (совпадает)
   
   - Название: "Юбка из замши — коричневый (XS-5, S-7, M-5)", Колонка G: 17
     → sizes: {XS: 5, S: 7, M: 5}, quantityOverride: не устанавливаем (5+7+5=17, совпадает)
   
   - Название: "Дублёнка длинная из тасканы — коричневый (образец)", Колонка G: 1
     → sample: true, quantityOverride: 1
   
   - Название: "Жакет из замши...", Колонка G: 10 (но размеров нет в названии)
     → quantityOverride: 10
   
   7.2. ЦЕНА ЗА ЕДИНИЦУ (колонка H):
   - Берём значение из колонки "Стоймость 1 ед $" (колонка H / индекс 7)
   - Это цена одной единицы товара в долларах США
   - Может быть числом (например, 180, 280, 550)
   - Если колонка H пустая → цена не устанавливается (прочерк, поле price отсутствует в JSON)
   
   ЛОГИКА:
   - Проверяем колонку H (Стоймость 1 ед $):
     * Если значение заполнено и является числом > 0 → устанавливаем price из колонки H
     * Если колонка H пустая или не число → поле price не устанавливается (прочерк)
   - Цена берётся из Excel для каждой позиции отдельно (историческая цена на момент поставки)
   
   ПРИМЕРЫ:
   - Колонка H: 300 → price: 300
   - Колонка H: 180 → price: 180
   - Колонка H: пусто → поле price отсутствует в JSON (прочерк)
   - Колонка H: 0 → поле price отсутствует в JSON (прочерк, т.к. должно быть > 0)
   
   7.3. СЕБЕСТОИМОСТЬ (колонка N):
   - Берём значение из колонки "Себестоимость с учётом карго" (колонка N / индекс 13)
   - Это себестоимость одной единицы товара в рублях (с учётом карго)
   - Может быть числом (например, 16484.62, 25948.48, 47106.84)
   - Если колонка N пустая → себестоимость не устанавливается (прочерк, поле cost отсутствует в JSON)
   
   ЛОГИКА:
   - Проверяем колонку N (Себестоимость с учётом карго):
     * Если значение заполнено и является числом > 0 → устанавливаем cost из колонки N
     * Если колонка N пустая или не число → поле cost не устанавливается (прочерк)
   - Себестоимость берётся из Excel для каждой позиции отдельно
   - Колонка K (Себестоймость) НЕ используется
   
   ПРИМЕРЫ:
   - Колонка N: 16484.62 → cost: 16484.62
   - Колонка N: 25948.48 → cost: 25948.48
   - Колонка N: пусто → поле cost отсутствует в JSON (прочерк)
   - Колонка N: 0 → поле cost отсутствует в JSON (прочерк, т.к. должно быть > 0)
   
   7.3. ВЗАИМОСВЯЗЬ КОЛИЧЕСТВА И ЦЕНЫ:
   - Итоговая стоимость = цена × количество (вычисляется в приложении, если нужно)
   - В Excel есть колонка "Итоговая стоимость $" (колонка I), но мы её не используем напрямую
   - price и quantityOverride используются для расчёта суммы позиции

8. ПАРСИНГ РАЗМЕРОВ ИЗ НАЗВАНИЯ:
   Примеры:
   - "Плащ из замши — коричневый (one size-5)" → sizes: {OneSize: 5}
   - "Юбка из замши — коричневый (XS-5, S-7, M-5)" → sizes: {XS: 5, S: 7, M: 5}
   - "Дублёнка длинная из тасканы — коричневый (образец)" → sample: true, quantityOverride: 1
   - "Жакет приталенный из меха пони в стиле 80-х — чёрный (XS-1, S-3)" → sizes: {XS: 1, S: 3}

9. МАППИНГ PRODUCTID:
   СХЕМА НАЗВАНИЙ:
   - В Excel полное название: "Название модели — цвет (размеры)"
   - В products.json поле name: "Название модели — цвет"
   - Для маппинга используем: "Название модели — цвет" (без размеров в скобках)
   
   АЛГОРИТМ:
   1. Берём "Наименование" из колонки C (индекс 2)
   2. Убираем часть в скобках (размеры) - всё, что в скобках после названия
      * Находим последнюю открывающую скобку "("
      * Убираем всё начиная с этой скобки до конца строки
      * Обрезаем пробелы с конца
   3. Нормализуем пробелы в названии:
      * Заменяем множественные пробелы на один пробел
      * Это важно для корректного сравнения (например, "меха  кёрли" → "меха кёрли")
      * Пример: "Дублёнка короткая из меха  кёрли — серый" → "Дублёнка короткая из меха кёрли — серый"
   4. Получаем "Название модели — цвет" для поиска
   5. Ищем в products.json продукт, где product.name (также нормализованный) точно совпадает с очищенным названием
   6. Если найден → берём product.id и используем как productId
   7. Если не найден → в каталог добавляется новый товар (id: auto-NNN, name: очищенное название, category: по названию см. ниже, остальные поля — минимальные); в позицию подставляется этот productId; после парсинга каталог сохраняется в data/products.json.
   
   Каталог data/products.json при каждом парсинге собирается заново: стартовый список пустой, в каталог попадают только товары из текущего Excel (по уникальным названиям). Категория у каждого выводится по названию: приоритет «питон» → Экзотика; иначе «кож» → Кожа; иначе «мех» → Мех; иначе «замш» → Замша; иначе «Прочее». Так «кожи питона» даёт Экзотику, меху приоритет выше замши.

   Размеры в каталоге (product.sizes) заполняются автоматически после парсинга: один проход по всем поставкам и rawItems, для каждого productId собираются все ключи из item.sizes (те же размеры, что уже парсятся из названия в разделе 8). Список размеров товара — объединение всех размеров, встречающихся у этого изделия в партиях. Порядок в product.sizes канонический: xs, s, m, l, xl, onesize; неизвестные размеры — в конце. Дополнительного парсинга Excel не требуется, используются уже посчитанные rawItems[].sizes.
   
   ВАЖНО: Нормализация пробелов применяется как к названию из Excel, так и к названиям из каталога products.json
   для обеспечения корректного сравнения даже при наличии лишних пробелов в исходных данных.
   
   ПРИМЕРЫ:
   Excel: "Жакет из замши в стиле 80-х — коричневый (one size-20)"
   → Очищаем: "Жакет из замши в стиле 80-х — коричневый"
   → Ищем в products.json: product.name === "Жакет из замши в стиле 80-х — коричневый"
   → Находим: { id: "suede-003", name: "Жакет из замши в стиле 80-х — коричневый", ... }
   → productId: "suede-003"
   
   Excel: "Юбка из замши — коричневый (XS-5, S-7, M-5)"
   → Очищаем: "Юбка из замши — коричневый"
   → Ищем в products.json: product.name === "Юбка из замши — коричневый"
   → НЕ НАЙДЕН → создаём новый товар (id: auto-001, name: "Юбка из замши — коричневый"), подставляем productId: "auto-001"
   
   Excel: "Жакет приталенный из кожи питона — бежевый глянцевый (XS-5, S-5)"
   → Очищаем: "Жакет приталенный из кожи питона — бежевый глянцевый"
   → Ищем в products.json: product.name === "Жакет приталенный из кожи питона — бежевый глянцевый"
   → Находим: { id: "python-004", name: "Жакет приталенный из кожи питона — бежевый глянцевый", ... }
   → productId: "python-004"
   
   Excel: "Дублёнка длинная из тасканы — коричневый (образец)"
   → Очищаем: "Дублёнка длинная из тасканы — коричневый"
   → Ищем в products.json: product.name === "Дублёнка длинная из тасканы — коричневый"
   → НЕ НАЙДЕН → создаём новый товар (id: auto-NNN), подставляем productId

10. СПЕЦИАЛЬНЫЕ ФЛАГИ ДЛЯ РУЧНОЙ ПРАВКИ:
   Следующие флаги НЕ парсятся из Excel и добавляются вручную в JSON после парсинга:
   
   - paidPreviously: true
     * Используется для позиций, которые были оплачены ранее (в другой поставке)
     * Влияет на сумму: позиция не учитывается в расчёте суммы поставки (sum = null)
     * Пример: {"status": "Получено, оплачено ✅", "paidPreviously": true, "note": "оплачен ранее"}
   
   - noPayment: true
     * Используется для позиций, которые не требуют оплаты (например, вернулись с ремонта)
     * Влияет на сумму: позиция не учитывается в расчёте суммы поставки (sum = null)
     * Пример: {"status": "Получено, не оплачено 📦", "noPayment": true, "note": "вернулись с ремонта"}
   
   - showStatusTag: true
     * Используется для принудительного отображения тега статуса, даже если нет текста заметки
     * Пример: {"noPayment": true, "showStatusTag": true, "note": "вернулись с ремонта"}
   
   ВАЖНО: Эти флаги добавляются вручную после автоматического парсинга Excel.
   Парсер НЕ должен пытаться определять эти флаги автоматически из Excel данных.

11. ЛОГИКА ОПРЕДЕЛЕНИЯ ОПЛАТЫ (TypeScript):
   
   Функция isPaidStatus(text) в lib/statusText.ts определяет, считается ли статус "оплаченным":
   
   - Если статус содержит "не оплачен" → НЕ оплачено
   - Если статус содержит "оплачен частично" или "частично оплачен" → НЕ оплачено
   - Если статус содержит "оплачен" (и не попал под предыдущие правила) → оплачено
   - Если статус НЕ содержит слова "оплачен" → НЕ оплачено
   
   ПРИМЕРЫ:
   - "Получено, оплачено ✅" → оплачено (содержит "оплачено")
   - "Получено, не оплачено 📦" → НЕ оплачено (содержит "не оплачен")
   - "Оплачен частично" → НЕ оплачено (содержит "оплачен частично")
   - "В производстве 🛠️" → НЕ оплачено (не содержит "оплачен")
   - "В пути 🚚" → НЕ оплачено (не содержит "оплачен")
   
   Эта логика используется:
   - На странице "Надвигающаяся расплата" для фильтрации партий и расчёта сумм
   - Для определения цвета суммы в интерфейсе (зелёный/акцентный)
   - Для сортировки групп позиций (оплаченные вниз)
